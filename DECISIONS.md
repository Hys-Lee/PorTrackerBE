# PorTracker BE 아키텍처 결정 이력 (DECISIONS.md)

본 문서는 PorTracker 백엔드 설계 및 구현 과정에서의 핵심 기술 의사결정(Architecture Decision Records, ADR)을 구체적인 배경, 라이브러리 선정 사유, 특정 필드 설계 배경 및 잠재적 이슈 극복 방안 등의 상세 디테일과 함께 기록합니다.

## 🏛️ Decision Log

### 💡 ADR-01. In-Memory SQLite 전환 및 표준 SQL 백업 명령어 채택
* **결정 사항**: 기존 로컬 디스크 파일 기반의 유저별 SQLite 구조(`db/{userId}.db`)를 프로세스 종료 시 소멸하는 In-Memory SQLite(`jdbc:sqlite:file:memdb_{userId}?mode=memory&cache=shared`) 구조로 전환하고, SQLite 표준 백업 명령을 백업/복원에 사용합니다.
* **이유 및 선정 배경**:
  * 512MB RAM 서버 인프라 환경에서 파일 I/O 누적 시 성능 저하가 발생하므로 Zero-Disk 아키텍처를 추구했습니다.
  * 또한 디스크에 유저 데이터가 남는 것 자체가 프라이버시 관점의 보안 리스크이므로 완전히 가상 메모리로 데이터를 격리합니다.
* **잠재적 이슈와 기술적 한계 극복**:
  * *이슈*: 초기 구현 시 자바 `unwrap(SQLiteConnection.class)` 및 `SQLiteConnection.backup/restore` 메서드를 사용하려 하였으나, 드라이버 버전에 민감하여 특정 환경에서 해당 메서드를 찾지 못해 컴파일 오류가 발생하는 강한 결합도가 존재했습니다.
  * *극복 방안*: 드라이버 클래스 캐스팅에 의존하지 않고, SQLite JDBC가 공식 지원하는 표준 SQL 명령어인 **`backup to 'file_path'`** 및 **`restore from 'file_path'`**를 `jdbcTemplate.execute()`를 통해 호출하도록 리팩토링했습니다. 이로 인해 컴파일 에러를 차단하고 코드 가독성 및 유지보수성을 극대화했습니다.
  * 덮어쓰기 백업 시 파일 충돌을 사전에 막기 위해 디스크 버퍼 생성 직전 `Files.deleteIfExists`를 명시적으로 태우는 세이프가드 코드를 내장했습니다.

### 💡 ADR-02. 이벤트 소싱(Event Replay) 기반 Kafka WAL 무손실 복구
* **결정 사항**: 인메모리 데이터의 휘발성을 방어하기 위한 WAL(Write-Ahead Log) 메커니즘으로 외부 Apache Kafka 브로커(`user-transaction-logs` 토픽)를 단일 진실 공급원으로 활용하고, 비즈니스 메서드 리플렉션을 통해 Replay(Redo) 복구를 처리합니다.
* **이유 및 선정 배경**:
  * 사용자 CUD 발생 시마다 Supabase RDBMS에 트랜잭션 쿼리를 직접 쏘면 쓰기 병목이 생깁니다.
  * Kafka는 고성능 Append-Only Log 구조이므로 메시지를 **동기식(Sync Send - Ack=all)**으로 전송해 물리 영속성을 확보한 뒤 유저에게 성공 응답을 주어도 RDBMS 직접 쿼리보다 압도적으로 빠르고 안전합니다.
* **잠재적 이슈와 기술적 한계 극복**:
  * *이슈 1. 무한 루프 발생*: 복구(Replay)를 위해 유저의 지난 비즈니스 메서드를 호출할 때, 이 호출이 다시 AOP에 감지되어 카프카에 중복 로깅을 유발하는 순환 호출(Infinite Loop) 위협이 있었습니다.
    * *극복*: ThreadLocal 플래그 관리 클래스 `ReplayContextHolder`를 신설하여, Replay 스레드가 기동하는 중에는 AOP의 로깅 로직을 안전하게 건너뛰도록(Skip) 방어막을 구축했습니다.
  * *이슈 2. 복구 순서 보장*: 분산 환경에서 변경 큐의 메시지가 역전되어 실행되면 복구 결과가 왜곡됩니다.
    * *극복*: 유저 식별자인 `userId`를 Kafka Message Key로 바인딩하여 카프카 내부에서 **동일한 유저의 이벤트는 동일 파티션에 무조건 FIFO(선입선출) 순서로 적재**되는 순서 보증 규격을 적용했습니다.
  * *이슈 3. 복구 범위 특정*: 매번 전체 카프카 메시지를 다 읽는 것은 비효율적이며, 카프카 자체의 컨슈머 오프셋(Consumer Offset)만 이용할 경우 카프카 서버 재구동 또는 메시지 보관 주기(Retention) 만료 시 GDrive DB 파일과의 동합성을 추적할 수 없는 한계가 있습니다.
    * *극복*: 데이터 스냅샷(GDrive)과 변경 내역(Kafka Offset)을 동일한 체크포인트 쌍으로 묶어 관리하기 위해, GDrive 업로드 성공 직후 최종 업로드된 카프카 오프셋 값인 **`last_sync_offset` (BIGINT)**을 Supabase 메인 DB의 유저 테이블에 영속 저장합니다. 복구 시 GDrive 파일 복원 후 `last_sync_offset + 1` 번 오프셋 메시지부터 안전하게 Replay를 개시하여, 카프카 인프라 상태에 무관하게 무결성을 100% 보장합니다.

### 💡 ADR-03. Supabase Profile 테이블 내 user_db_version 필드 추가 및 0ms Flyway 최적화
* **결정 사항**: Supabase 메인 DB의 `profile` 테이블에 **`user_db_version` (INTEGER)** 컬럼을 추가하고, 버전을 매핑하여 불필요한 런타임 Flyway 마이그레이션 검증을 생략(Skip)합니다.
* **이유 및 선정 배경**:
  * 인메모리 SQLite 특성상 사용자가 로그인해 접속할 때마다 매번 메모리에 DB가 올라가는데, 이때마다 Flyway가 데이터베이스 마이그레이션 유효성을 체크하는 것은 부팅 속도에 엄청난 병목이 유발됩니다.
* **필드 추가 및 극복 방안**:
  * 구글 드라이브에서 이미 마이그레이션이 완결된 `.db` 바이너리를 다운로드해 메모리에 복원하기 때문에, 평소에는 스키마 검증이 불필요합니다.
  * 서버의 환경 변수인 `APP_DB_VERSION`과 Supabase의 `user_db_version`을 1차로 대조하여, 일치할 경우 **Flyway 마이그레이션 모듈 자체의 실행을 건너뜀(Skip)**으로써 평소 접속 시 검사 성능 오버헤드를 **0ms**로 튜닝했습니다.
  * 신규 배포 등으로 앱의 스키마 버전이 변경되었을 때에만 딱 1회 Flyway 검증을 수행하고, 성공 즉시 Supabase의 `user_db_version`을 갱신 및 백업하여 다음 접속부터는 다시 스킵되도록 제어했습니다.

### 💡 ADR-04. Resilience4j Circuit Breaker & Spring Retry 라이브러리 선정
* **결정 사항**: 외부 네트워크 연동의 가용성 보장을 위해 Resilience4j 스타터 및 Spring Retry 라이브러리를 선정하여 통합했습니다.
* **선정 사유 및 현재/잠재적 이슈 극복**:
  * 구글 드라이브 API는 타사의 클라우드 서비스이므로 상시 연결 상태가 보장되지 않고, 네트워크 순단이나 속도 저하 시 백엔드 전체 스레드를 점유하는 잠재적 파괴력이 존재합니다.
  * **Resilience4j**: 스프링 부트 3 환경에서 서킷 브레이커 표준으로 사용하기에 가장 선언적이고 호환성이 높습니다. 구글 API 장애 감지 시 스레드 점식을 즉시 차단(Fail-fast)하고, 미리 설계된 `fallbackDownload` (Redis 캐시 스냅샷 반환), `fallbackUpload` (카프카 유예 로그 상태 알림)로 우회 처리하여 사용자 경험의 훼손을 차단합니다.
  * **Spring Retry**: 일시적 순단(Network Blink) 시 구글 백업이 완전 실패하는 것을 방지하고자, 일정 간격을 두고 3회 재시도합니다. 이때 구글 서버에 지속적인 부하 폭발(DDoS)을 막기 위해 **지수 백오프(`delay=2000, multiplier=2.0`)**를 결합하여 점진적으로 간격을 늘리며 재시도하도록 튜닝했습니다.

### 💡 ADR-05. Redis Fixed Window Counter Rate Limiter 구현
* **결정 사항**: Redis의 `increment`와 `expire` 단일 명령어를 활용한 OncePerRequestFilter 기반 유량 제어 필터를 신설하고 JWT 인증 직후에 바인딩했습니다.
* **도입 배경 및 극복 방안**:
  * 구글 드라이브 API는 분당 요청 제한(Rate Limit) 정책이 대단히 엄격하므로, 유저가 무한 클릭이나 루프 오류로 API를 연속 요청할 때 이를 차단할 세이프티 가드가 없었습니다.
  * Redis 분산 카운터 윈도우를 사용해 유저/IP당 분당 요청 횟수를 가볍게 누적하고, 100회 초과 시 즉시 `429 Too Many Requests` 상태코드를 응답하게 제어하여 구글 서버로 흘러가는 트래픽 흐름을 최전방에서 안전하게 방어했습니다.

### 💡 ADR-06. 통계 집계 성능 극대화를 위한 중간 집계 롤업(Rollup) 테이블 및 DB 트리거 도입
* **결정 사항**: 사용자의 대량 자산 거래 내역을 실시간으로 Sum/Average 쿼리하는 비용을 최적화하기 위해 통계 데이터용 **롤업(Rollup) 테이블**을 설계하고, DB 레벨의 **트리거(Trigger)**로 데이터를 점진적 증분 업데이트 처리합니다.
* **도입 배경 및 극복 방안**:
  * 유저의 거래 내역(Portfolio)이 누적될수록 실시간 통계 조회 쿼리는 시간복잡도가 $O(N)$으로 증가하여 병목이 됩니다.
  * 이를 극복하기 위해 통계 뷰에 직접 매핑되는 중간 집계 롤업 테이블(`group_statistic` 등)을 사전에 구축하고, 원장 데이터 수정 발생 시 데이터베이스 자체 트리거가 이를 감지하여 해당 항목만 미세 증분 연산($O(1)$)하도록 처리함으로써 백엔드 연산 부하를 제로화했습니다.

### 💡 ADR-07. 자산 조회 응답성 향상을 위한 Redis 2배수 선캐싱(Prefetching) 캐시 정책
* **결정 사항**: 조회 지연 최소화 및 쿼터 제한 준수를 위해 첫 화면에 노출되는 데이터 한도의 **2배수 분량을 Redis에 사전 캐싱**해 두고 프론트 선요청(Prefetch) 연동과 결합합니다.
* **도입 배경 및 극복 방안**:
  * 사용자 로그인 직후 혹은 포트폴리오 첫 로딩 시 Google Drive 로딩 대기 레이턴시(3s)를 유저가 직접 체감하는 문제를 방지하기 위함입니다.
  * 첫 노출 영역에 표시할 데이터의 2배 범위 자산 내역을 Redis 캐시 공간에 미리 적재(Prefetch)해 두어 사용자의 스크롤 다운이나 추가 클릭 시 지연 시간을 0ms로 대폭 단축하여 극단적인 UX 속도를 보증합니다.

### 💡 ADR-08. 데이터 및 자격 증명 보안을 위한 AES-256-GCM 양방향 암호화 도입
* **결정 사항**: Supabase에 저장되는 구글 OAuth 토큰(특히 Refresh Token) 및 Kafka에 발행하는 변경 이력(WAL) JSON 페이로드를 AES-256-GCM 알고리즘으로 양방향 암호화하여 적재하고, 사용할 때 복호화합니다.
* **배경 및 근거**:
  * *이슈*: 클라우드 데이터베이스(Supabase) 혹은 외부 카프카 브로커가 해킹 등으로 침해당할 경우 유저의 구글 드라이브 접근 권한이 무단 취득되거나 모든 자산 변경 트랜잭션이 외부에 완전히 유출되는 심각한 프라이버시 위험이 상존했습니다.
  * *극복 방안*: 대칭키 암호화 표준 중 보안 강도가 가장 높은 **AES-256-GCM**을 적용하고, 메시지 복호화 시 재전송 공격을 예방하기 위해 고유 IV(Initialization Vector)를 생성해 데이터 앞부분에 결합하여 저장합니다. 대칭 암호화 키는 어플리케이션 소스코드에 커밋하지 않고 환경 변수 `APP_ENCRYPTION_KEY`를 통해서만 주입되도록 설계했습니다.

### 💡 ADR-09. 컨테이너 침투 방지를 위한 Docker 비루트(Non-root) 샌드박스 실행
* **결정 사항**: Dockerfile 빌드 단계에서 `root`가 아닌 일반 사용자 계정(`portracker`)을 생성하여 스프링 부트 프로세스를 해당 사용자 권한으로 샌드박싱 구동합니다.
* **배경 및 근거**:
  * *이슈*: 기존 Dockerfile 구조에서는 빌드 및 프로세스가 root 계정 하에서 가동되었습니다. 이 경우 만약 어플리케이션 취약점으로 인해 컨테이너 원격 코드 실행(RCE)이 뚫리거나 컨테이너 이스케이프(Container Escape) 취약점이 터지면 호스트 OS 전체 권한이 탈취되는 보안 재앙을 초래합니다.
  * *극복 방안*: 최소 권한의 원칙(Principle of Least Privilege)을 적용하여 컨테이너 환경을 일반 계정으로 샌드박싱함으로써, 백엔드가 침해되더라도 컨테이너 밖으로 권한을 전파할 수 없도록 OS 레벨 격리를 완료했습니다.

### 💡 ADR-10. SpEL 동적 키 바인딩 및 waitTime=0 설정을 통한 Redisson 분산락 중복 요청 방지 고도화
* **결정 사항**: 기존의 정적이고 임시적인 Redisson 분산락 키 파싱 방식을 Spring Expression Language (SpEL) 기반 동적 파서(`CustomSpringELParser`)로 전면 개편하고, `waitTime = 0` 방식을 지원하여 단시간 중복 API 호출(더블 클릭 등)을 최전선에서 즉각적으로 무산(Fail-fast)시킵니다.
* **이유 및 선정 배경**:
  * 기존 `@DistributedLock` 어노테이션 기반 락은 락 키가 하드코딩되거나 파라미터 인덱스에만 의존하여 유연하지 못했고, 락 대기 시간(waitTime)이 기본값(5초)으로 잡혀 있어 유저가 따닥 더블 클릭을 했을 때 두 번째 요청이 대기 큐에 걸렸다가 결국 순차적으로 수행되어 트랜잭션이 중복 처리되는 잠재 정합성 결함이 있었습니다.
* **잠재적 이슈와 기술적 한계 극복**:
  * *이슈 1. SpEL 해석 결합*: 런타임 시 AOP의 joinPoint 정보로부터 메서드 매개변수 명과 실제 인수를 해석하여 유저 ID 등을 추출해내야 합니다.
    * *극복*: `StandardEvaluationContext`에 메서드 파라미터 맵을 주입하여 SpEL 해석을 대행하는 `CustomSpringELParser`를 제작했습니다. 이로 인해 `@DistributedLock(key = "'sync:' + #userId")` 처럼 실무적으로 매우 유연하고 완결성 높은 동적 키 구성이 가능해졌습니다.
  * *이슈 2. 중복 클릭 즉각 차단*: 중복 요청 방지 시 대기 시간 없이 즉시 429 혹은 예외를 반환해야 합니다.
    * *극복*: `waitTime = 0` 설정을 부여하면 락을 선점한 즉시 다음 연타 요청은 락 시도 즉시 실패(`available = false`) 처리되며, 대기 큐 점유 없이 즉각 `BusinessException(ErrorCode.TOO_MANY_REQUESTS)` 에러를 내려줌으로써 불필요한 트랙잭션 발생 및 서버/DB 리소스를 완벽히 보호하였습니다.
